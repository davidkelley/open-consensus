# Open Consensus — Implementation Plan

## Context

The user runs a hand-rolled `/consensus` Claude Code command: an orchestrating agent fans a plan out to several other CLI coding agents (different models = a diverse expert panel), collects their critiques, and iterates until the panel agrees. It produces noticeably better plans, but the mechanism is fragile: agents fail to respond, the orchestrator hangs waiting, retried/garbage replies bloat the orchestrator's context window.

**Open Consensus** turns that pattern into a robust standalone application. It is an *execution substrate* (a "proxy") that the driving agent calls through an **MCP tool**. The driving agent stays the orchestrator — it decides when consensus is reached and composes follow-up prompts — but it offloads the messy mechanics (spawning CLIs, parallelism, timeouts, retries, failure isolation, response distillation) to Open Consensus. Users manage their agents and panels, and watch consensus unfold in a live terminal timeline, through an ink + React CLI/TUI.

Shipping this enables: (1) any agent (Claude Code, Codex, …) to drive a reliable multi-agent consensus loop from a single MCP tool, without polluting its own context; (2) users to declaratively configure a roster of CLI agents and group them into task-specific panels; (3) graceful degradation when an underlying CLI is down, slow, uninstalled, or unauthenticated.

> **This plan has been hardened by one round of the `/consensus` reviewer panel (Codex + opencode returned full reviews; the Grok CLI was down — 403 spend limit — and that reviewer fell back to its own reasoning).** All accepted round-1 findings are folded in below; rejected ones are listed at the bottom.

## Core architecture (load-bearing decisions)

```
                 ┌─────────────────────────────────────────────┐
   Orchestrator  │  MCP server (stdio, spawned by Claude Code)  │
   (Claude Code) │  thin client → forwards to daemon            │
        │  MCP   └───────────────┬─────────────────────────────┘
        └───────────────────────▶│  unix socket (0600) [default] / loopback HTTP+token (fallback)
                                  ▼
                 ┌─────────────────────────────────────────────┐
                 │  Daemon (long-lived, single-instance)        │
                 │   • Engine: runs → rounds → invocations      │
                 │   • Process runner: spawn / tree-kill / caps │
                 │   • Concurrency, timeout, retry, quorum      │
                 │   • SQLite (metadata + event log) + raw blobs│
                 │     on disk; persisted event cursor          │
                 └───────────────┬───────────────┬─────────────┘
                                 ▼               ▼
              execFile, shell:false, own process group,        SSE (snapshot + Last-Event-ID replay)
              prompt via stdin/temp-file, read-only flags        │
              claude / codex / gemini / opencode CLIs            ▼
                                              ┌──────────────────────────────┐
                                              │ TUI (ink+React, claude-code-  │
                                              │ style /slash-commands + live  │
                                              │ timeline; separate process)   │
                                              └──────────────────────────────┘
```

**D1 — Runtime/stack.** TypeScript (`strict`) on Node ≥20 (have 24.14), with **`zod` at every boundary** for runtime validation (D17). ink + React (a claude-code-style slash-command TUI — D19), `@modelcontextprotocol/sdk` (MCP), `commander` (non-interactive CLI parsing), `vitest` (tests, **≥90% coverage gate** — D18), **Biome** (formatting + linting, replacing ESLint/Prettier), `tsup`/esbuild (build). JS-native throughout because ink and the MCP SDK are.

**D2 — A persistent local daemon is the source of truth.** A consensus run spans many minutes and outlives any single MCP tool call or TUI session, so a single-instance daemon holds engine state and orchestrates CLI execution; the MCP server and TUI are thin clients. **Default transport is a unix domain socket with `0600` perms**; the socket lives in a short runtime dir (`$XDG_RUNTIME_DIR` or `/tmp`) with a short name to stay under the ~104-byte `sun_path` limit (with a fallback if the resolved path is too long). Loopback HTTP with a random port + bearer token is a fallback, and when used it **validates `Origin`/`Host` by exact-match string compare** (DNS-rebinding protection, no prefix matching) and binds `127.0.0.1` only — never `0.0.0.0`. The token lives in a `0600` file in the state dir, not in an env var. The daemon writes its resolved **endpoint** (socket path, or a dynamically OS-assigned loopback port — never a fixed port, to avoid `EADDRINUSE`) plus token to a `0600` discovery file under the runtime dir, which the MCP server and TUI read to connect — written **atomically** (temp + rename) so a reader never sees a partial endpoint, with clients **retrying under a short bounded wait** ("daemon starting…") rather than connecting to a stale/half-written value (NTFS can't enforce `0600`; documented as a Windows-uncertified limitation). The stdio MCP server **auto-starts** the daemon if absent (see D14).

**D3 — The orchestrator owns the consensus *judgment*; Open Consensus owns *execution*** (prompt dispatch, adapter process management, result persistence). It fans a round out to every panel agent, isolates failures, and returns *distilled, provenance-tagged* per-agent responses. The orchestrator reads them, decides whether the panel has converged, and either composes the next round's prompt or finalizes. Automatic semantic "consensus reached" detection is a v1 non-goal.

**D4 — Orchestrator↔daemon = async job model with snapshot long-poll.** `consensus_start` returns a `runId`+`roundId` immediately (non-blocking). The orchestrator then calls `consensus_poll(runId, roundId, wait_ms)`: a bounded long-poll that returns the moment the round reaches a terminal state, else after `wait_ms`. Properties that make it robust:
- **Snapshot + state version:** every poll returns a monotonic `stateVersion` and the full current round snapshot; polls are idempotent and safe to repeat.
- **Tiny "still running" payload:** an in-progress poll returns only counts/statuses (no transcripts), so even several polls cost the orchestrator almost no context.
- **Bounded `wait_ms` with a concrete ceiling:** the daemon enforces a max wait (default 50s, configurable) and the MCP server caps any client-supplied `wait_ms` to it, keeping each long-poll comfortably under typical MCP host tool timeouts so the *number* of poll calls (hence context entries) stays minimal. If a host's own tool timeout still fires first, the orchestrator is told (tool docs + `next_action`) to treat a tool error/timeout as **transient** and simply re-poll the same `runId/roundId` — safe because polls are idempotent and the run keeps progressing inside the daemon regardless.
- **Connection errors are retryable:** if the daemon restarts during a long-poll the tool returns a structured `transient`/re-poll signal, not a hard failure — the orchestrator re-polls after backoff.
- Retries/backoff happen **inside the daemon** and are never replayed to the orchestrator (only summarized as attempt counts). This is the core fix for "claude code stops waiting" + context bloat. SSE (D11) serves the TUI and is *optionally* offered to Streamable-HTTP MCP clients, but **long-poll is the contract**.

**D5 — Rounds are stateless per agent in v1, but the schema is session-aware.** Each round, every agent receives a fully-composed prompt carrying all needed context; we do **not** rely on per-CLI session/resume in v1 (inconsistent across tools). The data model carries `sessionMode: stateless | resume` and a per-adapter `sessionId` field from day one so resume can be added later without a rewrite (`stateless` is the only v1 behavior; a `fork` mode is intentionally deferred to avoid shipping undefined semantics). The orchestrator is responsible for keeping each round's composed prompt bounded; adapters enforce a max prompt size and **prefer delivering the prompt via stdin or a temp file** to avoid `E2BIG` and process-listing leakage; an adapter whose CLI only accepts a positional prompt argument falls back to argv but is then bound by a stricter size cap, with the residual `ps`-visibility documented (the prompt itself is not secret — secrets travel via env, never argv).

**D6 — Response distillation is a concrete, structured contract.** Full raw output is captured **to a file on disk** (not a giant SQLite blob — see D9), keyed by `runId/roundId/agentId/attempt` so each retry's output is retained and retrievable (the default `rawRef` points at the final attempt and can enumerate earlier ones). The MCP result returns a **structured** per-agent object: `{ status, distilled, attempts, durationMs, errorClass?, truncated, rawRef }` where:
- `distilled` is produced by the adapter's parser: it **prefers the CLI's JSON/structured output mode** to extract the agent's final answer in full; only when no structured mode exists does it fall back to ANSI-stripping and taking the last *complete* paragraph(s) up to a byte cap, appending an unambiguous marker `…[truncated: N bytes omitted; rawRef=<id>]…`. When `truncated` is set, the per-agent `next_action` advises the orchestrator to `consensus_get_raw` before judging (so code-producing agents whose code precedes their summary aren't misjudged on a truncated tail).
- The byte/line cap is an explicit, documented constant (configurable per panel).
- `rawRef` lets the orchestrator fetch full output later via `consensus_get_raw` (D12) — paginated by **byte-offset cursor** into the on-disk blob, with a default page size and a **hard per-call max-bytes cap** so a caller can't pull an unbounded transcript into the model's context — without bloating context by default.
- **Distillation is purely deterministic** (structured-output extraction or ANSI-strip + truncate); it is **never an LLM call**, so the daemon introduces no hidden token spend.

**D7 — Config model (validated, snapshotted).**
- **Agent** = `{ id, name, adapter (cli tool id), model?, args[]: string[], env{}: record, timeoutMs, maxRetries, sessionMode? }`.
- **Panel** = `{ id, name, agentIds[], quorum, concurrency?, roundDefaults }`.
- Stored as validated JSON under an XDG-resolved config dir (with a documented note that on macOS we deliberately use XDG paths, not `~/Library`). **Each run snapshots the resolved config** (agents, adapter versions, resolved binary paths, env-allowlist digest, prompt template) into SQLite so round-to-round comparisons and history are auditable even if config changes mid-run (D15).

**D8 — Adapters are pluggable, capability-typed, and read-only by default.** An `Adapter` declares: `id`, `detect()` (binary present + parseable `--version`), `capabilities` (non-interactive mode? structured-output mode? read-only/sandbox flag? prompt via stdin vs arg?), `buildInvocation(prompt, cfg)` → `{ file, args[], stdin|tempFile, env }`, and `parse(stdout, stderr, exitCode)` → structured result. **Semantic classification (`ok` / `refusal` / `error`) is the adapter's responsibility**, using adapter-specific knowledge (documented exit codes, a JSON envelope, or a structured tail) — the process runner only supplies the exit code plus cleaned streams and never guesses intent from a 0/non-0 split. Concrete v1 invocations (verified against installed `--help`, pinned, defensive):
- `claude` → `claude -p/--print` (+ `--output-format json` when available), **read-only permission mode / no edit tools by default**.
- `codex` → `codex exec` in **read-only sandbox** mode.
- `gemini` → `gemini -p` non-interactive.
- `opencode` → `opencode run`.
- A deterministic **mock adapter** (configurable ok/slow/timeout/error/refusal) plus a **"chaos child" fixture binary** (spawns grandchildren, emits ANSI + >1 MB output, ignores SIGTERM) used to test the process runner. Adapters whose required non-interactive interface is absent are marked **unavailable** (no PTY/`expect` driver in v1).

**D9 — Persistence: SQLite for metadata + event log, raw blobs on disk.** `better-sqlite3` is synchronous (runs on the main thread), so to avoid blocking the event loop during a multi-agent round we keep SQLite rows small (run/round/invocation metadata, distilled text, the event log) and stream **raw CLI output to capped files on disk** with incremental flush + an in-memory buffer cap (spill, never unbounded). WAL mode; retention/pruning policy (D16). (Node 24's built-in `node:sqlite` is noted as an alternative if `better-sqlite3` native-build friction appears.)

**D10 — Process runner is a hardened security/lifecycle boundary, not a convenience wrapper.** Every invocation: `execFile`/`spawn` with `shell:false` and **argv arrays only** (prompt never interpolated into a shell string or argv); child placed in **its own process group** (`detached:true` + retained stdio pipes for stdin prompt delivery); termination goes through a **`ProcessTerminator` abstraction** — a POSIX backend (`kill(-pgid, SIGTERM)`→`SIGKILL`) and an *uncertified* Windows backend (`taskkill /T /F` or a Job Object, since `process.kill(-pgid)` does not exist on Windows) — so a cancel/timeout **reaps the whole tree** on the certified platforms and degrades rather than crashes elsewhere; stdout/stderr read with **byte caps + backpressure** (spill to the raw file, never buffer unbounded); **ANSI/control sequences stripped** before distillation/persistence; **secrets redacted before anything is written** (env-value patterns + common key formats like `sk-…` in stderr) — opt-in only for unredacted raw capture; the raw exit code + cleaned streams are handed to the adapter, which owns the `{ ok, refusal, error }` classification (D8). **Prompt temp files** are written `0600` in a private per-run dir and unlinked once the child has consumed them (and swept on reconcile), so prompt contents never linger. If a child exceeds the **output byte cap** it is tree-killed and marked terminal `error` (output-overflow). The **orphan sweep** on daemon start is *scoped*: every spawned process group is tagged with a sentinel env var (`OPEN_CONSENSUS_DAEMON=<id>`) and recorded in a PGID registry in the state dir; the sweep is driven by the **persisted PGID registry** (authoritative — no brittle process-table/`ps` scanning by env var; the sentinel is only a defensive cross-check before signaling), killing only recorded groups whose PGID is **still alive and matches the recorded process start-time/identity** (so a recycled PID now owned by an unrelated process is never signalled), never unrelated user processes; the registry carries a `schemaVersion` and an incompatible state dir is treated as ephemeral. The sweep runs on daemon start, periodically, and on cancel (not start-only), so a delayed restart can't leave tagged groups running unboundedly; the runner also unlinks its prompt temp file in a `finally` block so an abnormal child exit still cleans up, with reconcile as the backstop.

**D11 — SSE and long-poll are two *views* over one persisted event log, not two state machines.** All engine events are appended to the SQLite event log with monotonic sequence numbers. Long-poll (D4) reads a snapshot at a `stateVersion`; SSE (`/events`) is a live tail the TUI uses, supporting `Last-Event-ID` so a reconnecting client back-fills missed events from SQLite. One source of truth, two consumers (model vs human). A snapshot is a **complete authoritative state** (never a delta): it always suffices to reconstruct the current run/round/agent state without having seen any intervening events, which is what makes `snapshot-reset` safe when a reconnecting client's `Last-Event-ID` has been pruned.

**D12 — MCP tool surface (explicit, foolproof).** `consensus_list_panels`, `consensus_list_runs({state?})` (**re-entry point**: lets a restarted/replaced orchestrator re-anchor to in-flight runs and resume judgment), `consensus_start({panel, prompt, options, idempotencyKey?})` (creates run + first round), `consensus_round({runId, prompt, agentIds?, idempotencyKey?})` (adds a round to an existing run; requires a valid `runId`; an `idempotencyKey` makes a retried call return the existing round instead of duplicating it), `consensus_poll({runId, roundId, wait_ms})`, `consensus_status({runId})`, `consensus_cancel({runId})`, `consensus_cancel_agent({runId, roundId, agentId})`, `consensus_get_raw({rawRef, cursor?, maxBytes?})` (**paginated** — never returns an unbounded payload; the orchestrator pages with `cursor` and stops at EOF). Poll/status results include a structured **`next_action` hint** (`keep_polling | review_results | start_next_round | finalize | handle_degraded`) so the driving model reliably executes the start→poll→decide→round loop, and **any** MCP call referencing a `runId` (poll/status/round/get_raw) resets that run's idle-reaper TTL, so an orchestrator reasoning quietly can heartbeat with `consensus_status` and won't be auto-parked. **Idempotency is durable and scoped:** keys are persisted in SQLite alongside run state; the lookup predicate is `(runId, idempotencyKey)` only (a replayed call's `prompt`/`agentIds` are ignored — the stored round wins) and a match returns the **original** result rather than erroring or duplicating (if the matched round is still in-flight, the replay returns the current snapshot — exactly as a `consensus_poll` would — never an error or a duplicate round), surviving daemon restarts for the run's lifetime + a grace TTL. Idempotency records are scoped to their run and **pruned with it** in the coordinated retention sweep (D16) so the table never grows unbounded; `consensus_list_runs` and any history result is **paginated** so a hot call never loads or deep-validates an unbounded set. Tool descriptions teach the loop and are **adversarially tested against the real orchestrator model**.

**D13 — Round completion & quorum semantics (precise).** A round is **complete only when every agent has reached a terminal state** (`ok | refusal | timeout | error | unavailable | cancelled`) — quorum is a *validity verdict*, not an early exit, because consensus wants every panel member's input. `quorum` counts **only `ok` (non-partial) responses**. Verdict: `met` (≥ quorum ok), `degraded` (some ok but < quorum), `failed` (0 ok). The poll result **always lists missing/errored agents by name with their error class** so the orchestrator never mistakes "2 of 4 ok, strongest reviewer never ran" for success. Liveness is guaranteed by the **per-agent timeout**: every invocation reaches a terminal state (a hang → tree-kill → `timeout`), so the round verdict (`met`/`degraded`/`failed`) is computed the instant the last invocation goes terminal — a round never hangs waiting, and this does **not** depend on the idle-run reaper (which only handles an orphaned *orchestrator*, never in-flight liveness).

**D14 — Daemon lifecycle & auto-start race.** Single-instance is enforced by an **atomic OS lock** — an `flock`/`O_EXCL` advisory lock on a well-known lockfile (carrying PID + start-time for diagnostics) that must be **acquired before any startup reconcile or prune-repair runs** — closing the read-PID-then-decide TOCTOU so two daemons can never both "reclaim" a stale lock and reconcile concurrently. A stale lock whose owner is provably dead is reclaimed atomically. On auto-start, the loser of a start race **waits for the socket to become ready** (bounded retry/backoff) rather than spawning a second daemon; the MCP server only returns once the daemon answers a health check. A run is treated as **orphaned by its orchestrator** and auto-parked (run marked `abandoned`) only when it is **idle** — no in-flight round with actively-progressing invocations — **and** its `runId` has gone untouched by any MCP call for a configurable TTL (sane default ~15 min, with an enforced minimum floor so a misconfig like `0` can't abandon everything); a round still producing output is never reaped, any poll/status/round/get_raw resets the TTL, and `consensus_list_runs` surfaces parked runs for re-adoption. This stops a crashed/replaced orchestrator from leaking running CLI work without killing legitimately long-running silent agents. On Windows (uncertified) the socket is a **named pipe** with an equivalent ACL; loopback+token is the last-resort fallback.

**D15 — Crash recovery is *reconcile + re-dispatch*, not resume.** External CLI children cannot be resumed after a daemon restart (the OS has reaped them). Startup **reconcile** is a single authoritative repair pass: (a) the scoped orphan sweep kills any leftover tagged process groups; (b) every invocation left `running` whose group is gone → terminal `interrupted` (the process is dead; it is **never silently re-issued**, avoiding double-invocation), and its round's verdict is **recomputed** from the now-terminal invocations so no round is left stuck `running` — re-dispatching a fresh attempt is an explicit orchestrator-driven action (a new round), not a hidden automatic retry; (c) leftover `0600` prompt temp files in per-run dirs are deleted; (d) a **prune-repair** step reconciles any row↔blob mismatch from a crash mid-prune (orphaned raw blobs deleted; a `rawRef` whose blob is missing is marked `raw-missing` rather than dangling). "Resume the same process" is explicitly *not* claimed; the state machine and tests are written around this.

**D16 — Resource & retention limits.** Global + per-run concurrency caps; **per-tool serialization option** (default: serialize concurrent invocations of the *same* adapter to avoid races on shared CLI state in `~/.claude`, `~/.codex`, etc., while parallelizing across *different* tools); output byte caps; retention + pruning + `VACUUM`. **Pruning is coordinated**: a run's raw-blob files are deleted in the same sweep that prunes its SQLite rows (rows first would dangle `rawRef`s; blobs first would orphan files), and an SSE client whose `Last-Event-ID` predates the retained event window receives a **`snapshot-reset`** event telling it to reload a fresh snapshot rather than back-fill. An optional **disk-usage ceiling**, when hit, makes `consensus_start` return a structured capacity error and triggers an emergency prune of the oldest completed runs. So that prune can still run when the volume is near-full, the daemon keeps a small **pre-allocated reserve file** it deletes to free space for the prune's WAL/DELETE, and uses a journal mode tolerant of low free space. HTTP body-size limits and connection timeouts bound request resource use.

**D17 — `zod` at every boundary (type + runtime safety).** Every untrusted or cross-process boundary is parsed through a `zod` schema whose inferred TS type is the single source of truth: config files, MCP tool inputs *and* structured outputs, daemon HTTP request/response bodies, SSE event payloads, adapter `parse()` results, and the persisted run/round/invocation records (validated on read, so a schema migration or hand-edit can't silently corrupt state). Schemas live next to the types they describe and are shared by producer and consumer; `tsconfig` is `strict` and CI fails on any type error. **Persisted records carry a `schemaVersion`** and are read through **explicit, ordered SQLite migrations** that upcast old rows before validation; stored-record schemas are lenient (additive/unknown fields tolerated via `passthrough`) so a version mismatch never crashes startup or drops data. To keep `better-sqlite3`'s synchronous reads off the event-loop critical path, deep `zod` parsing is applied to **bounded** payloads (single records, small/paginated lists, untrusted external input); large raw blobs and big result sets are validated **structurally / at the metadata level**, not deep-parsed element-by-element, and internally-written rows are validated **on write** so reads stay light.

**D18 — Testing tiers (≥90% coverage, with isolated live E2E).** Two explicitly separated tiers:
- **Default (`npm test`)** — vitest over **mock + chaos-child + fake-binary** fixtures only: deterministic, no real CLIs, no network, no spend. Enforces a **≥90% line+branch coverage gate** (vitest `coverage.thresholds`, CI-failing). To keep the gate *honest* rather than gamed: hard-to-drive process/signal code (tree-kill, SIGTERM→SIGKILL escalation, orphan sweep, `flock`, low-disk prune) is covered **meaningfully** by the chaos-child/fake-binary harness, and the few genuinely non-deterministic defensive branches are excluded with a **lint-enforced justified** comment requiring a reason (an ESLint rule, with a CI grep as backstop, rejects bare/unjustified ignores) — never a hollow mock that hits the line without exercising the behavior. The everyday + PR gate.
- **Live E2E (`npm run test:e2e:live`)** — a separate, **opt-in** suite driving the full stack against **real agent CLIs** (`claude`, `codex`, …), **mechanically isolated**: it lives in its own `test/e2e-live/` dir under a dedicated vitest config (the default config's globs never discover it), runs only under an explicit `OPEN_CONSENSUS_E2E_LIVE=1` guard, and the **default suite asserts that env var is unset and hard-fails if it is set** — so it can never run or spend by accident. It **skips cleanly** when required CLIs/auth are absent, **caps its own agent concurrency**, and serves as the **final release quality-gate**, asserting a real multi-CLI consensus completes and degrades gracefully when one CLI is killed mid-round.

**D19 — The TUI is a claude-code-style slash-command app, on a layered seam.** `open-consensus` with no subcommand launches a full-screen ink + React interactive app modeled on Claude Code / Codex / opencode: a **two-region** layout — finalized lines render once via ink `<Static>` (append-only, never re-rendered), while the **in-progress run timeline renders in a separate dynamic region that updates in place** (because `<Static>` cannot be mutated), with events **coalesced/throttled** to stay smooth under high SSE rates and the run's **full finalized output committed to `<Static>` on completion in a single atomic transition** (not a lossy summary; no line dropped/duplicated if an SSE event lands mid-handoff) — a persistent bottom prompt with **`/command` autocomplete + history**, and inline **live consensus timelines** from daemon SSE. Commands: `/help`, `/agents`, `/agent add|edit|remove|test`, `/panels`, `/panel create|edit|add-agent|set-quorum`, `/runs`, `/run <panel> <prompt>`, `/run view <id>`, `/daemon status`, `/quit`. **Two layers, not one:** `command-core` is a **stateless** request/response client (config ops + daemon RPC + zod validators) shared verbatim by the `commander` CLI *and* the TUI; a separate **`tui-session`** layer owns the long-lived concerns the CLI never has — SSE subscription lifecycle (exponential-backoff+jitter reconnect, cap ~30s), transcript state, autocomplete corpus, and input focus. The CLI imports only `command-core` (so it never hangs on a dangling SSE/stream at exit); the TUI imports both. The boundary is **enforced statically in CI** by a dependency-cruiser rule that fails the build (workspace-package separation is a noted future hardening) so `command-core` can never accidentally pull in SSE/`AbortController` lifecycle and silently re-leak. **`Ctrl+C` semantics:** when a run/action is in flight it cancels that action — the client `AbortController` aborts the request *and* a `consensus_cancel`/`cancel_agent` RPC tells the daemon to **tree-kill the in-flight child** (so nothing keeps running server-side) — and prints a `cancelled` line once the daemon **acks** the cancel (bounded ~500 ms wait so a fast exit can't lose the RPC); a **second `Ctrl+C` escalates** — the TUI directly `SIGKILL`s the known child PGID (bypassing a wedged daemon) and exits — so two Ctrl+Cs always terminate. When idle, a single Ctrl+C exits gracefully. Priority is deterministic: ink raw-mode `useInput` handles the keypress while the app is mounted, and a process `SIGINT` handler is only the fallback when raw mode isn't active; it never kills the daemon. The `tui-session` SSE reconnect **re-reads the endpoint discovery file on every attempt** (the daemon may have restarted on a new dynamic port), treating a momentary `ENOENT` during the daemon's atomic rename as transient (retry), never caching a dead URL.

**D20 — Read-only execution is best-effort, not a sandbox — and said so loudly.** Panel agents are full coding CLIs that *can* edit files and reach the network; an adapter flag cannot *guarantee* read-only. So each adapter (a) uses the tool's strongest native constraint — permission/sandbox/no-tools/`--allowedTools`-none / read-only mode — as its **default**; (b) runs every invocation in an **ephemeral scratch working directory** (never the user's repo/cwd) to prevent *relative-path* accidents. **(b) is explicitly NOT a security boundary**: a confused or malicious agent given an absolute path, or making network calls, retains full user-level filesystem + network access — user-facing docs must state exactly this and must **never** call the scratch dir "isolation". (c) Each adapter records a `sandbox` capability; tools that can't be natively constrained are **elevated-opt-in only, behind an explicit acknowledgment** ("this tool can read/write/exfiltrate anything your account can reach"). True OS-level jailing (bwrap / Docker / seatbelt) is the only real containment and is out of v1 scope (noted).

**D21 — First-run UX, CLI auto-detection, and MCP registration.** Out of the box `consensus_list_panels` must not be empty: `open-consensus init` (run on first launch / explicitly) **auto-detects installed agent CLIs** (`which`/`where` + `detect()` across the registry, showing what was found and letting the user confirm/override so an nvm/asdf/Homebrew-shadowed binary isn't silently mis-picked) and **seeds a sensible default panel** + agent entries, writing a documented example config. `open-consensus mcp install` registers the MCP server for Claude Code / Codex **safely**: it parses the target host config first, **detects conflicts/existing entries** (idempotent — a matching entry is a no-op), writes **atomically** (temp + rename), and — when targeting multiple hosts — prints a **per-host success/failure summary** and exits non-zero on any failure (the default targets a single canonical host; multi-host is explicit), with a paired `mcp uninstall` to cleanly remove entries; it never corrupts a malformed host config, and explains telling the orchestrator to "use Open Consensus". When the daemon isn't running, the CLI/TUI **auto-start** it (shared logic) or emit a clear, actionable error — never a silent hang.

## Assumptions

- Each target CLI supports a non-interactive "one prompt → print → exit" mode (`claude -p`, `codex exec`, `gemini -p`, `opencode run`); exact flags + structured-output support are verified per-adapter against the installed binary's `--help` and pinned. Adapters degrade to **unavailable** if the expected interface isn't found; no PTY/`expect` driver in v1.
- Target OS is macOS/Linux first (XDG paths; Windows tree-kill via `taskkill` is implemented but not certified in v1).
- The user has the relevant CLIs installed and authenticated; Open Consensus does not manage their auth. Missing/unauthed/auth-expired-mid-run are **handled failure modes**, not hard errors.
- Concurrent same-tool invocations may race on shared CLI state/auth caches → mitigated by per-tool serialization (D16); cross-tool runs parallelize.
- Each invocation can cost money and hit the network → **tests never invoke real adapters**; adapter behavior is covered by fixtures + a spawnable fake-binary (no spend), and the process runner by the chaos-child fixture.
- Daemon, MCP server, and TUI run on the same machine as the orchestrator; no remote/cloud transport in v1; single user, single-instance daemon.

## Non-goals

- ACP integration (we drive CLIs directly).
- Cloud/remote/multi-tenant daemon, hosted web UI, or auth beyond the local socket/loopback token.
- Automatic semantic "consensus reached" detection / auto-merging plan content (orchestrator decides in v1).
- Managing the underlying CLIs' own auth/installation.
- Stateful per-CLI session resumption across rounds (schema-ready, but off in v1).
- A PTY/`expect` driver for REPL-only tools; Windows certification.

## Failure scenarios (modeled explicitly)

| Scenario | Handling |
|---|---|
| CLI not installed / not on PATH | `detect()` fails → agent `unavailable`; round still valid if quorum met; reported by name. |
| CLI installed but unauthed / auth expires mid-run | non-zero exit or interactive auth prompt → timeout kills tree → terminal `error`, isolated; reported with error class. |
| CLI hangs / too slow | per-agent timeout → **process-group tree kill** → bounded retries → terminal `timeout`. |
| CLI edits files / runs commands during "review" | adapters default to **read-only/no-edit/sandbox** flags; elevated execution is explicit opt-in. |
| Panel below quorum | verdict `degraded`/`failed` with per-agent reasons; orchestrator decides (poll `next_action: handle_degraded`). |
| Concurrent same-tool runs race on `~/.claude` etc. | per-tool serialization (D16); cross-tool parallelism preserved. |
| Config drift mid-run | resolved config snapshotted per run; availability re-checked at round start; unavailable agents reported, not fatal. |
| Daemon crash mid-run | reconcile-on-start: advance stuck `running` rounds→terminal/re-dispatch, scoped orphan sweep, prune row↔blob repair, temp-file sweep; **no false resume**. |
| Orchestrator long-poll times out / daemon restarts during poll | snapshot poll returns `still running` / `transient`; orchestrator re-polls; no hang, no context bloat. |
| Huge / ANSI-laden CLI output | byte caps + backpressure + spill to raw file; ANSI stripped; distilled+capped to MCP; full via `consensus_get_raw`. |
| Secrets in env / stderr | redacted **before** persistence/logs/SSE/TUI; unredacted raw capture is opt-in only. |
| Two daemons started simultaneously | lockfile (PID+start-time); loser waits for readiness; stale locks reclaimed. |
| Grandchild/orphan processes | own process group + tree kill + *scoped* startup orphan sweep (sentinel-tagged groups only). |
| Orchestrator crashes/restarts mid-run | `consensus_list_runs` re-anchors a replacement orchestrator; idle-run reaper auto-parks runs no one is polling so CLI work never leaks. |
| Agent output exceeds byte cap | child tree-killed, terminal `error` (output-overflow); partial raw retained on disk. |
| Disk-usage ceiling reached | `consensus_start` returns a structured capacity error + emergency prune of oldest completed runs. |
| Prompt temp files | `0600` in a private per-run dir, unlinked after the child reads them, swept on reconcile. |
| Agent ignores read-only flag and writes files | every invocation runs in an isolated ephemeral scratch cwd (D20) — writes land in a throwaway dir, not the user's repo; residual risk documented. |
| Loopback port already in use | daemon binds a dynamic OS-assigned port (never fixed) and publishes endpoint+token to a `0600` discovery file the clients read (D2). |
| Idempotency / history tables grow unbounded | idempotency + run records pruned together in the coordinated retention sweep; list/history APIs paginated (D12/D16). |
| Stored record from an older schema version | `schemaVersion` + ordered migrations upcast on read; lenient `passthrough` parsing so an upgrade never drops data or crashes startup (D17). |
| Empty config on first run | `open-consensus init` auto-detects installed CLIs + seeds a default panel so the orchestrator isn't handed an empty roster (D21). |
| Windows (uncertified) process termination | `ProcessTerminator` abstraction: POSIX PGID backend on macOS/Linux, `taskkill /T /F`/Job Object on Windows; Windows is best-effort/uncertified (D10). |
| User cancels mid-run from the TUI | Ctrl+C aborts the client request **and** sends a daemon cancel RPC that tree-kills the in-flight child — nothing keeps running server-side (D19). |
| Agent given an absolute path / network access | scratch cwd stops relative-path accidents only — **not** a security boundary; residual full-FS/network risk documented; uncontainable tools are elevated-opt-in behind an explicit acknowledgment (D20). |

## Phase 3 — final cross-stage review (dispositions)

**Applied:**

Full per-config-change historical auditability is a deferred enhancement, not a correctness bug.
- **D11/D16 SSE `snapshot-reset` (opencode high, Gemini low).** Moot under the implemented retention model: pruning is **whole-run** (`pruneRun` deletes a run + all its events), not a global sliding event window. A reconnecting client's viewed run either still exists (events intact, back-fill complete) or was pruned entirely (gone) — there is no "partial window" in which a timeline is silently incomplete, so the reset signal can't arise.
- **D5/D10 `tempFile` prompt delivery.** The frozen adapter contract advertises `promptDelivery`/`ctx.promptFile` for forward-compat, but **no v1 adapter uses it** (claude/codex/mock = stdin; gemini/opencode = arg). The engine implements the delivery modes adapters actually use; temp-file delivery is a deferred capability with no current consumer.
- **D16 disk-usage ceiling / emergency prune / reserve file.** Explicitly **optional** in the plan ("An *optional* disk-usage ceiling…"). Deferred; coordinated per-run pruning (rows + raw blobs) IS implemented.
- **D21 multi-host `mcp install`.** The implementation IS the planned **default** ("the default targets a single canonical host; multi-host is explicit"). Aggregated multi-host summary/partial-failure is the explicit extension, not the default.
- **D10 orphan-sweep PID identity.** A **documented Stage-5 design decision**: the detached-process-group sweep "accepts the (rare) pgid-reuse race" (see `daemon.ts`), trading start-time identity tracking for simplicity. Reaffirmed.
- **D17 SSE event-payload deep validation.** The SSE producer is the daemon itself (trusted, internally constructed); the TUI decoder guards on `type` and drops malformed frames rather than crashing. Deep zod on every frame conflicts with D17's "reads stay light"; the untrusted boundaries (config, MCP inputs, HTTP bodies, adapter `parse`) ARE validated.
- **D19 SSE event coalescing/throttling (Gemini low).** The per-event reducer is pure + cheap and ink batches renders; explicit throttling is a deferred perf optimization, not a correctness issue.
